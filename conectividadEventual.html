Análisis de conectividad eventual


Fase de pruebas iniciales:

En la ventana principal podemos encontrar los primeros errores evidentes en cuanto a la conectividad eventual. Luego de instalar la aplicación, podemos observar que, al abrir esta sin una conexión estable de internet wifi o datos móviles (simulados activando el modo avión del dispositivo), la pestaña de descubrir (ventana inicial que se abre en el dispositivo cuando se abre por primera vez) no puede cargar los podcasts básicos de muestra de iTunes. Así, se despliega un error que es poco informativo al usuario final y no respeta la falta de conocimiento técnico del usuario. El error antes mencionado como se puede observar en las fotos anteriores es “Unable to resolve host “itunes.apple.com”: No address associated with hostname”. En caso de no existir una conexión estable, el objeto incluido no debería desplegar este error sino un aviso que le informe al usuario el motivo por el cual la información no esta disponible de manera amigable. De esta forma se le puede informar si no dispone de una conexión wifi o una conexión de datos estable. De este modo el usuario que tiene poca experiencia puede revisar si su problema es de cobertura celular, o inclusive que tiene el wifi apagado.

Inicialmente, podemos observar que el tab de Cola no presenta ningún problema, pues ninguna de sus funciones requiere una conexión estable. En el tab de Episodios vemos que se maneja bien la información, pues al no disponer de ningún podcats almacenado se informa al usuario la ausencia de estos en español y de manera clara. La falta de internet en este caso no afecta la navegabilidad de la pestaña. Las descargas manejan el caso donde no existe ningún podcast disponible, así que no se ve afectado realmente por la conectividad eventual. El historial de reproducciones muestra un logo de “Sin historial” por lo que se informa al usuario el problema o más bien estado actual de la aplicación y no se ve afectado en ningún caso por la falta de conectividad de la aplicación. El tab de suscripciones únicamente nos permite navegar a añadir podcast donde encontramos todos los problemas iniciales que ya hemos mencionado (cuando no se tienen ningún podcast agregado).

Si se continua sin conectividad, la búsqueda local funciona bien (Buscar Podcast), sin embargo, al añadir una URL en Añadir pódcast por URL y dar en continuar se produce un error: Host Desconocido (Unable to resolve host “XXXX”: No address associated with hostname. No solo se da un manejo bastante inadecuado al mostrar el error completo al usuario (error generado por defecto) sino que además no se muestra en el mismo idioma las diferentes partes del error. Con esto, vemos que la aplicación tiene un diseño muy básico y muy poco enfoque en posibles problemas generados. Al presionar el botón de más en Descubrir encontramos el error java.net.UnknownHostException: Unable to resolve host etc. Nuevamente se hace evidente que se lanza la excepción de la aplicación sin capturarla y manejarla de acuerdo con las posibles interacciones del usuario final. Esto mismo ocurre al presionar Búsqueda avanzada, donde si se selecciona Buscar en Itunes, Explorar en gpodder.net o Buscar en fyyd, se genera también la misma excepción java mencionada anteriormente. Todo lo anterior se presenta en el video adjunto para que se pueda observar de primera mano el mal manejo de esta ventana.


Fase de pruebas con Podcast (Suscripción):

Al realizar una suscripción podemos observar que tanto en la pestaña de Episodios como suscripciones aparece ya disponible el podcast (o álbum de podcast) al que nos hemos suscrito. Al presionar descargar en cualquiera de las dos pestañas, el celular reconoce que no tiene conexión a wifi; sin embargo, se asume de inmediato que el usuario si dispone de datos móviles. Así, procede a preguntar al usuario si le permite realizar la descarga por medio de datos móviles. Al aceptar esto, sale un letrero que evidencia que comenzó la descarga (Mensaje mostrado en un Toast cuando se debió utilizar un SnackBar), pero como tampoco se dispone de este tipo de conexión se informa por medio de una notificación nativa que la descarga ha fallado. Aun cuando se informa de esta manera (Manera incorrecta de hacerlo), no se le explica al usuario el porque de la falla en la descarga. Es decir, nunca se avisa al usuario que no dispone de una conexión a wifi o a internet por medio de datos móviles. Esto se evidencia en el video adjunto. Las demás ventanas presentan un comportamiento igual al mencionado en la fase inicial, pues realizar una suscripción no afecta de manera directa el contenido o la respuesta de alguna de estas.


Fase de pruebas con Podcast (Descargado):

Al realizar la descarga de un podcast y luego entrar nuevamente a modo avión, se puede observar en los tabs de añadir podcast, historial de reproducciones y suscripciones los patrones de ejecución que se describieron en la fase inicial se repiten. En este caso, al observar la pestaña de episodios, se puede observar que se muestra un álbum con la información de los diferentes podcasts y solo se muestra disponible para la reproducción el que específicamente fue descargado. Al realizar click en los demás se evidencia el mismo problema que ocurría en la fase dos, donde el dispositivo reconocía que no se encontraba descargado e intentaba realizar la descarga a pesar de no tener una conexión estable a internet. Adicionalmente el fallo lo notifica por medio de push notifications. La otra opción a elegir redes móviles para realizar la descarga es agregar a la cola para poder realizar la descarga mas adelante. En este caso esta función no requiere intervención en cuanto a conectividad eventual, pues al quedar en cola el dispositivo puede reanudar o realizar la descarga cuando se recupere la conexión exitosamente.


Reporte Especifico con detalle técnico:

A lo largo de la investigación de las 3 fases se pudieron identificar varios problemas de conectividad eventual como fue mencionado. A continuación, se presentará uno a uno los errores con su tipo de falla y la forma como se debería dar un manejo adecuado para arreglar la falla. Primero se presentarán las fallas encontradas en la fase inicial donde no se perdía la conexión justo después de instalar la aplicación. Luego la segunda fase donde se realizaba una suscripción y por ultimo la fase donde se descarga un podcast y se observa el comportamiento general de la aplicación.

Claramente se puede observar que la aplicación utiliza un mecanismo Pull, pues la información solo se recibe cuando el usuario solicita descargar o visualizar un podcast nuevo. Esto es verdad para toda la información disponible excepto para la sección de Descubrir, donde se utiliza “Launch-Time”, pues es necesario mostrar al usuario la información de posibles podcast sugeridos sin que este solicite más información. De esta forma se atrae al usuario a buscar posible información que puede llegar a ser de su interés. Dada la naturaleza de la aplicación y la lógica de negocio que la rige, únicamente se utilizan estos dos tipos de “Data Fetching”.

En este se informa un error que además de ser en inglés, podría dar al usuario la idea de que el problema se encuentra en el destino y no en la falta de conectividad del celular. De esta forma el usuario podrá identificar el problema en su celular o su entorno y solucionarlo rápidamente para continuar utilizando el celular.


El primero error consiste en confirmar el envío para añadir un podcast por URL. Al realizar un envío vacío (sea con o sin internet) se evidencia el error anterior. Nuevamente se evidencia claramente un “Message with Exception Trace”. Así, no captura apropiadamente el error y el usuario final no logra entender cual es el problema con la solicitud que ha realizado, pues no se explica con detalle porque sucedió el error. En especial porque lo que entiende el usuario es que se esperaba algo en el índice 7, lo cual no tiene ninguna relevancia al usuario.


Este problema se debe solucionar realizando una verificación previa al realizar clic en confirmar. En caso de que el campo de entrada del usuario se encuentre vacío, se debería informar a este que debe llenar una URL antes de realizar el envío. Así, no se debería permitir que este avance a la siguiente actividad, y mas bien informar al mensaje correcto al usuario para que este pueda realizar las correcciones pertinentes (ingresar un URL valido).

Este problema se debe solucionar utilizando el catch y capturando la excepción correctamente. También se le debe informar que revise la URL ingresada y lo intente nuevamente para poder continuar exitosamente. En lugar de avanzar a la siguiente actividad, se debería revisar si se dispone de una conexión estable a internet y en caso de que no exista, se avise al usuario que revise su conexión a internet pues no hay una red estable para realizar la petición solicitada.



Los 3 errores anteriores se producen cuando se realiza una Búsqueda avanzada ya sea por Itunes, por gpodder.net o por fyyd. En este caso las 3 opciones generar el mismo error, donde en este caso es un “Message with Exception Trace”. Así, se muestra literalmente el mensaje de excepción generado desde el código java “UnknownHostException”. Al tener en cuenta que esto se genera al momento de perder la conexión a internet, solucionar este problema es relativamente sencillo. Una ves se hace clic en búsqueda avanzada se debería verificar si se dispone de una conexión estable a la red de datos o wifi. Así, no se debería permitir que el usuario avance a la actividad de búsqueda, sino que solo debería aparecer un mensaje informativo y claro que le permita al usuario saber que no posee internet. Así, este puede realizar las correcciones necesarias en su dispositivo o en su entorno para poder continuar con el uso adecuado de la aplicación. Al tener una conexión estable estos mensajes no aparecen y se redirige correctamente a los respectivos servicios para realizar la descarga correctamente.


Fase 2:

Esta fase se investigo con el fin de encontrar posibles errores previos a una descarga. Así, buscamos encontrar posibles funcionalidades perdidas, o funcionalidades no permitidas luego de restablecer la conexión a datos. Para lograr esto se realizó una suscripción (una de las funciones principales de la aplicación) a un canal de podcasts así, el historial del usuario debía existir, pero no reproducir, pues no se selecciono realizar la descarga del contenido. Con el fin de replicar estas pruebas se debe realizar la suscripción con conexión a datos móviles o wifi disponibles y luego activar el modo avión para simular la perdida total de conexión. Además de las mismas fallas que la fase anterior, algunos de los problemas identificados se presentan a continuación:


Al reproducir el contenido con conexión y luego presentar una pérdida de conexión simplemente se detiene la reproducción, pero no se avisa nada al usuario sobre el motivo de la parada de reproducción espontanea. De alguna forma este error puede ser clasificado como una funcionalidad perdida en la categoría de contenido perdido, pues el usuario nunca es notificado sobre la perdida de conectividad y los botones de reproducción dejan de funcionar con naturalidad. Igualmente se evidencia el problema claro de imagen vacía, pues, aunque inicialmente se había cargado una lista de 3 imágenes, al intentar ver la segunda y tercera sin conexión únicamente se muestra el espacio vacío.


El primero problema mencionado debe ser solucionado utilizando un método que reconozca el momento en que ocurra una falla en la conectividad y el contenido en reproducción no se encuentre disponible. En este caso se debe notificar al usuario para que reestablezca su conexión o permitirle volver para reproducir otro contenido que si se encuentre descargado actualmente. El segundo error mencionado debe ser solucionado utilizando un cache temporal que guarde la información del objeto (1 podcast con toda la información que ello implica como imágenes, artista y colección). Así, aunque exista una falla en la conectividad igualmente el usuario puede navegar la información de la aplicación por lo menos para registrar cual es el podcast que estaba escuchando y así poder reproducirlo más tarde.


Este error se origina al intentar realizar una descarga de un podcast que se encuentra dentro de las suscripciones. En este intento se despliega un permiso para utilizar redes móviles, pues se detecta que no se tiene conexión al Wifi previamente. Al autorizar el uso de redes móviles, no se verifica que con esta si se tenga conexión al wifi disponible, por lo que se informa al usuario que la descarga ha comenzado y luego se anuncia la notificación de fallo en la descarga como se evidencia en la foto anterior.


Para arreglar el problema anterior de debería realizar que existe una conexión estable a datos luego de que el usuario acepte utilizar redes móviles. Así, si no existe conexión disponible se debe suspender la descarga y no avisar al usuario que se encuentra realizando una descarga en ese momento. Así, se esperaría que el usuario revisara su conexión y luego se pudiera reintentar la descarga por realizar.


Este error se puede observar cuando se va a la pestaña de descargas luego de intentar realizar una descarga de la aplicación. Claramente se puede ver que cuando se selecciona la pestaña de Registro en Descargas se muestran los errores encontrados en los intentos realizados. En este caso, podemos ver que el mensaje que se agrega al layout es muy genérico. Así, aunque se captura la excepción correctamente, no se guarda el mansaje apropiado para evidenciar que la descarga se detuvo por perdida de conectividad en el dispositivo del usuario.


Para solucionar esta falla se debe capturar la excepción generada correctamente y en lugar de especificar un mensaje genérico agregar el motivo de la falla. En este caso se podría explicar que el motivo se debió a falta de conectividad cuando el usuario se encontraba realizando la descarga. Adicionalmente se debería implementar un reintento de descarga para que el usuario una ves sea notificado pueda volver a realizar la descarga del Podcast.


Fase 3:

En esta fase únicamente se evidencian los problemas anteriores, pues al tener un podcast descargado las opciones diferentes a realizar son escucharlo sin necesidad de conectividad esto ocurre con normalidad y las fallas en conectividad eventual realmente no afectan de la misma manera que se afectaban las dos fases anteriores. Lógicamente se encuentran los mismos errores antes reportados, pero al reproducir el contenido descargado tanto el audio como el texto y las fotografías funcionan correctamente.
